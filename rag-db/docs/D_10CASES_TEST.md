## 概述

本测试完全仿照 `test_runner.run_category_d()` 的**端到端模式**：

```
口语化输入 → QueryOptimizer → VectorStore → Reranker → DeepSeek LLM
```

对每个用例**分解5个环节**，逐一记录：
- **运行耗时 (ms)**：每个环节的独立计时
- **准确率贡献**：优化器标准化是否有效、检索科室是否匹配、LLM置信度

### 与完整 test_runner D组 的区别

| 维度 | test_runner D组 | test_d_10cases |
|------|:---:|:---:|
| 用例数 | 50+ | **10** (精选) |
| 对比维度 | 优化 vs 非优化 | **5环节耗时分解** |
| 输出重点 | LLM对比 | **各组件准确率+耗时** |
| 逐组件计时 | 无 | **有** (opt/search/rerank/compr/llm) |

4. **代表性**：每个类别选最典型的输入表达式

## 测试流程 (5环节分解)

每个测试用例依次经过5个环节，每个环节独立计时：

```
┌─────────────────────────────────────────────────────────────────┐
│ [环节1] QueryOptimizer.optimize()                               │
│   输入: "肚子疼拉稀想吐没胃口" (口语化)                           │
│   输出: "腹痛 腹泻 恶心 食欲不振" (标准化)                        │
│   耗时: ~1ms (rule模式, 词典匹配)                                │
│   准确率: 是否提取出有效症状?                                     │
├─────────────────────────────────────────────────────────────────┤
│ [环节2] VectorStore.search_disease()                            │
│   输入: 标准化后的 query                                         │
│   输出: Top-10 候选疾病 (cosine similarity)                      │
│   耗时: ~10-30ms (BGE-M3 embedding + ChromaDB)                  │
│   准确率: Top-1 的 cosine 分数                                   │
├─────────────────────────────────────────────────────────────────┤
│ [环节3] Reranker.rerank_results()  (如模型可用)                  │
│   输入: query + 10个候选文档                                     │
│   输出: 重排序后的结果 (cross-encoder score)                      │
│   耗时: ~50-500ms (取决于 GPU/CPU)                               │
│   准确率: 排序是否变更? 分数Δ?                                    │
├─────────────────────────────────────────────────────────────────┤
│ [环节4] VectorStore.comprehensive_search()                      │
│   输入: 标准化 query + reranker (if enabled)                     │
│   输出: 综合结果 + primary_recommendation                        │
│   耗时: ~10-200ms (含3个Collection)                              │
│   准确率: 推荐科室是否匹配预期? ← 检索准确率判定                   │
├─────────────────────────────────────────────────────────────────┤
│ [环节5] DeepSeekClient.recommend_department()  (如API可用)       │
│   输入: 原始 query + Top-5 疾病候选                              │
│   输出: 科室推荐 + 疾病 + 置信度 + 推理 + 建议 + 紧急警告          │
│   耗时: ~1000-5000ms (LLM API)                                  │
│   准确率: LLM 置信度%, Token消耗, 紧急信号检出                    │
└─────────────────────────────────────────────────────────────────┘
```

**说明**:
- `Optimizer`: QueryOptimizer 词典匹配耗时 (通常 < 1ms)
- `Search`: ChromaDB + BGE-M3 向量检索耗时
- `Reranker`: Cross-Encoder 精排耗时 (0 = 模型未加载)
- `Compr`: comprehensive_search 综合耗时 (含3 Collection)
- `LLM`: DeepSeek API 网络+推理耗时 (0 = API未配置)

```

│   (判定标准: optimize后 symptoms 非空 且 optimized_query ≠ 原始输入)

│ Reranker 可用:                1/10
│ Reranker 排序变更率:          7/10
```

**准确率判定标准**:

| 组件 | 判定方式 | 通过条件 |
|------|----------|----------|
| Optimizer | `optimize()` 后 `symptoms` 非空 | 至少提取1个标准症状 |
| Search | `comprehensive_search` 科室匹配预期 | `预期科室 in 推荐科室` |
| Reranker | 排序变化 + 分数Δ | score_delta > 0 有利于 |
| LLM | `recommend_department` 返回有效置信度 | confidence > 0 且无 error |

## 与完整 test_runner 的关系

```
test_runner.py
├── A组 (100用例) — 本地检索准确率/latency
├── B组 (60用例)  — LLM推荐质量/Token
├── C组 (80用例)  — 优化前后对比
└── D组 (50用例)  — 优化+LLM端到端
                          │
                          │  仿照 D组模式
                          │  从 COMPREHENSIVE_TEST_CASES 取10条
                          │  增加 5环节逐组件计时
                          ▼
              test_d_10cases.py  ← 本文件
```

**何时使用 test_d_10cases.py**：
- 快速诊断：10个用例跑完 < 1分钟 (不含LLM) 或 3-5分钟 (含LLM)
- 瓶颈定位：表3直观显示哪个环节最慢
- 组件评估：表2给出每个组件的准确率贡献

**何时使用完整 test_runner.py**：
- 正式测试报告：需要大样本统计显著性 (>50 用例)
- 全部分类覆盖：需要18个身体系统 + 方言 + 边界全面数据
- 生成图表：需要 `--chart` 输出9种可视化图表
