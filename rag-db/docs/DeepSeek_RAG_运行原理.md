# RAG 向量数据库 + DeepSeek 生成层 — 运行原理详解

> 项目：智能医疗导诊系统  
> 技术栈：ChromaDB（向量存储）+ BGE-M3（嵌入模型）+ DeepSeek API（生成层）  
> 数据源：OpenKG 中文疾病知识图谱（8,808 条疾病记录）

---

## 一、整体架构概览

本系统是一个典型的 **RAG（Retrieval-Augmented Generation，检索增强生成）** 架构，核心流程为：

```
用户自然语言症状 → 查询优化 → 向量检索 → DeepSeek LLM 推理 → 结构化科室推荐
```

```
┌──────────────────────────────────────────────────────────────────┐
│                        RAGPipeline                               │
│                                                                  │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────────┐   │
│  │ QueryOptimizer│───>│ VectorStore  │───>│ DeepSeekClient   │   │
│  │ (口语标准化)  │    │ (ChromaDB)   │    │ (LLM 生成推理)   │   │
│  └──────────────┘    └──────┬───────┘    └────────┬─────────┘   │
│                             │                     │              │
│                      ┌──────▼───────┐      ┌──────▼─────────┐   │
│                      │   BGE-M3     │      │  DeepSeek API   │   │
│                      │ (1024维嵌入) │      │ (deepseek-v4)   │   │
│                      └──────────────┘      └────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

| 组件 | 作用 | 技术 |
|------|------|------|
| QueryOptimizer | 口语/方言 → 标准医学术语 | DeepSeek LLM + 规则词典 |
| VectorStore | 语义相似度检索 | ChromaDB + HNSW 索引 |
| DeepSeekClient | 基于检索结果的推理生成 | DeepSeek API（OpenAI 兼容） |

## 二、DeepSeek 作为生成层的角色定位

### 2.1 为什么需要 DeepSeek？

向量检索只能返回**相似疾病列表**，但无法完成以下任务：

| 向量检索能做的 | 需要 DeepSeek 才能做的 |
|---------------|----------------------|
| 返回 Top-5 匹配疾病 | 综合分析 Top-5，确定**最可能的单一诊断** |
| 给出相似度分数 | 给出**置信度评分**（0-100），并解释依据 |
| 输出疾病名称 | 生成**通俗易懂的推理链**，让患者理解 |
| 列出关联科室 | 判断是否有**紧急情况**（胸痛、大出血等） |
| | 提供**就医建议**和**备选科室** |

### 2.2 DeepSeek 被调用的两个环节

DeepSeek 在整个 Pipeline 中**被调用了两次**：

```
用户输入 "肚子疼拉稀想吐没胃口"
    │
    ▼
┌─────────────────────────────────────┐
│ 第1次 DeepSeek 调用                  │
│ QueryOptimizer._optimize_with_llm() │
│ 作用：口语标准化                      │
│ "肚子疼拉稀想吐没胃口"               │
│    → "腹痛 腹泻 恶心 食欲不振"        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ VectorStore.comprehensive_search()  │
│ 使用标准化后的文本进行向量检索         │
│ 返回 Top-5 匹配疾病                  │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 第2次 DeepSeek 调用                  │
│ DeepSeekClient.recommend_department()│
│ 作用：基于检索结果生成推荐            │
│ 输入：原始症状 + Top-5 检索结果       │
│ 输出：JSON 结构化推荐                 │
└─────────────────────────────────────┘
```

## 三、向量数据库层：ChromaDB 的运作机制

### 3.1 数据存储结构

ChromaDB 中维护 **3 个 Collection**，各自承担不同角色：

```
medical_rag_db/                     ← ChromaDB 持久化目录
├── chroma.sqlite3                  ← 元数据存储
├── 938f01a4-.../                   ← Collection UUID (disease_knowledge)
│   ├── header.bin                  ← HNSW 索引头
│   ├── data_level0.bin             ← 第0层向量数据
│   ├── length.bin                  ← 向量长度
│   └── link_lists.bin              ← HNSW 分层图链接
└── 263abfab-.../                   ← Collection UUID (symptom_dept_direct)
    └── ...
```

| Collection | 条目数 | 向量维度 | 用途 |
|-----------|--------|---------|------|
| `disease_knowledge` | 8,808 | 1024 | 主知识库：疾病+症状+科室 |
| `symptom_dept_direct` | 4,826 | 1024 | 症状→科室直接映射 |
| `department_info` | 54 | 1024 | 科室诊疗范围 |

### 3.2 文本向量化过程（Embedding）

使用 **BGE-M3 模型**（1024 维，约 2.2GB）将文本转为向量：

```
文本："疾病：感冒。症状：鼻塞、流涕、喷嚏、咳嗽..."
         │
         ▼  BGE-M3 编码（本地推理，无需联网）
         │
向量：[0.023, -0.341, 0.567, 0.128, ..., -0.089]  ← 1024 个浮点数
```

**关键设计**：向量化文本是**拼接后的结构化字符串**，包含疾病名、症状、科室、分类、简介五个字段，让嵌入模型一次性捕捉多维语义：

```python
# build_knowledge_base.py:115-121
search_text = (
    f"疾病：{name}。"
    f"症状：{symptom_text}。"
    f"所属科室：{dept_text}。"
    f"分类：{cat_text}。"
    f"简介：{desc_short}"
)
```

### 3.3 HNSW 近似最近邻搜索

ChromaDB 使用 **HNSW（Hierarchical Navigable Small World）** 算法进行向量检索：

```
        第2层（跳跃层，稀疏图）
    ● ──────────────── ●
    │                   │
    │    第1层（中间层）  │
    │  ● ── ● ── ●     │
    │  │ \  / │  │     │
    │  ● ─ ●  ●  ●     │
    │    第0层（基础层，稠密图）
    │  ●──●──●──●──●   │
    └──────────────────┘

搜索过程：
  1. 从第2层的入口点开始
  2. 贪心搜索：每一步跳向与查询向量更近的邻居
  3. 到达局部最优点后，下降到第1层继续搜索
  4. 最终在第0层精确搜索 Top-K 最近邻

复杂度：O(log N) ≈ log(8808) ≈ 13 次比较（而非暴力遍历 8808 次）
```

### 3.4 相似度度量：余弦距离

```
余弦相似度 = cos(θ) = (A · B) / (|A| × |B|)

ChromaDB 存储的是 cosine distance = 1 - cos(θ)
代码中转换回相似度：score = 1 - distance

当 θ = 0°  → score = 1.00 → 语义完全相同
当 θ = 30° → score = 0.87 → 高度相似
当 θ = 60° → score = 0.50 → 中等相似
当 θ = 90° → score = 0.00 → 无关
```

### 3.5 两跳推理机制

系统从症状到科室的推荐经过**两次跳转**：

```
第一跳（语义检索）：
  "头痛发热咳嗽" ──向量相似度──> "感冒" (score: 0.746)
                                  "流行性感冒" (score: 0.704)
                                  "急性上呼吸道感染" (score: 0.698)

第二跳（结构化映射）：
  "感冒".cure_department → ["内科", "呼吸内科"]
  "流行性感冒".cure_department → ["内科", "呼吸内科"]
  "急性上呼吸道感染".cure_department → ["呼吸内科"]

综合推荐：呼吸内科（多个疾病共同指向）
```

这种设计将**语义理解**（第一跳）和**精确映射**（第二跳）解耦，避免了纯 LLM 可能产生的"幻觉科室"。

## 四、DeepSeek 生成层：从检索结果到结构化推荐

### 4.1 API 调用方式

DeepSeek 使用 **OpenAI 兼容接口**，通过标准 `openai` Python SDK 调用：

```python
# deepseek_client.py:64-67
self.client = OpenAI(
    api_key=self.api_key,          # 从 .env 读取
    base_url="https://api.deepseek.com",  # DeepSeek API 端点
)
```

| 配置项 | 值 | 说明 |
|--------|-----|------|
| API 地址 | `https://api.deepseek.com` | 兼容 OpenAI 格式 |
| 默认模型 | `deepseek-v4-flash` | 快速、高性价比 |
| 推理模型 | `deepseek-v4-pro` | 复杂推理场景 |
| 温度参数 | `0.3` | 低温度保证输出稳定性 |
| 最大 Token | `600` | 控制回复长度 |
| 输出格式 | `response_format: json_object` | 强制 JSON 结构化输出 |

### 4.2 Prompt 工程：系统提示词设计

DeepSeek 的推荐能力依赖于精心设计的 **System Prompt**：

```
┌─────────────────────────────────────────────────┐
│               System Prompt 结构                 │
├─────────────────────────────────────────────────┤
│ 1. 角色定义："你是一个智能医疗导诊助手"            │
│ 2. 输出格式：严格的 JSON Schema 定义              │
│ 3. 约束规则：                                     │
│    - department 必须是知识库中出现的科室           │
│    - 多结果指向同一科室时提高优先级                │
│    - confidence 四级阈值指南                       │
│    - 危急症状检测规则                             │
│ 4. 可解释性：推理依据用通俗语言                    │
└─────────────────────────────────────────────────┘
```

**置信度四级阈值设计**：

| 置信度 | 含义 | 动作 |
|--------|------|------|
| ≥ 80 | 症状与知识库高度匹配 | 直接推荐 |
| 60-79 | 基本匹配，有一定不确定性 | 推荐 + 建议补充症状 |
| 40-59 | 部分匹配 | 建议详细描述 |
| < 40 | 匹配度低 | 建议人工分诊 |

### 4.3 检索上下文格式化

Top-5 检索结果被格式化为结构化上下文，注入 User Message：

```python
# deepseek_client.py:108-115
context_parts = []
for i, r in enumerate(rag_results[:5]):
    context_parts.append(
        f"{i+1}. {r['disease']} (相似度: {r['score']:.1%})\n"
        f"   症状: {r['symptoms']}\n"
        f"   科室: {r['departments']}\n"
        f"   分类: {r['category']}"
    )
```

最终发送给 DeepSeek 的消息结构：

```
┌──────────────────────────────────────────┐
│ System Message                           │
│ - 角色定义 + JSON 格式要求 + 约束规则     │
├──────────────────────────────────────────┤
│ User Message                             │
│ - ## 患者症状描述                         │
│   用户原始输入                            │
│ - ## 知识库检索结果 (Top-5)               │
│   1. 感冒 (相似度: 74.6%)                │
│      症状: 鼻塞, 流涕, 喷嚏...            │
│      科室: 内科, 呼吸内科                 │
│   2. 流行性感冒 (相似度: 70.4%)          │
│      ...                                 │
└──────────────────────────────────────────┘
```

### 4.4 DeepSeek 输出与后处理

DeepSeek 返回结构化 JSON，代码进行**防御性解析**：

```json
{
  "department": "呼吸内科",
  "disease": "感冒",
  "confidence": 85,
  "reasoning": "患者症状头痛、发热、咳嗽、流鼻涕与感冒高度匹配...",
  "suggestion": "建议前往呼吸内科就诊，多饮水，注意休息...",
  "alternative_departments": ["内科", "急诊科"],
  "emergency_warning": false
}
```

**异常处理链**：

```
DeepSeek 返回
    │
    ├── JSON 解析成功 ──> 标准化 confidence 为整数 ──> 返回结果
    │
    ├── JSON 解析失败 ──> 尝试从文本提取 ──> 失败则返回兜底结构
    │
    └── API 调用异常 ──> 返回 {"department": "服务异常", ...}
```

### 4.5 Token 消耗

每次推荐的典型用量：

| 指标 | 数值 |
|------|------|
| Prompt Tokens | ~580 |
| Completion Tokens | ~120 |
| 总计 | ~700 tokens/query |

按 DeepSeek 定价，每次推荐成本约 **¥0.001**（千分之一元）。

## 五、查询优化层：DeepSeek 辅助的口语标准化

### 5.1 为什么需要查询优化

用户输入与知识库术语之间存在**语义鸿沟**：

```
用户习惯表达               知识库标准术语
─────────────              ─────────────
"肚子疼"         ──X──>   （无法直接匹配）
"拉稀"           ──X──>   "腹泻"
"打摆子"         ──X──>   "寒战"
"心慌"           ──X──>   "心悸"
```

### 5.2 三种工作模式

| 模式 | 机制 | 优点 | 缺点 |
|------|------|------|------|
| **LLM 模式** | 调用 DeepSeek 进行智能标准化 | 覆盖广、理解上下文 | 依赖网络、有延迟 |
| **规则模式** | 600+ 条口语→标准词典匹配 | 离线可用、毫秒级 | 覆盖有限 |
| **混合模式**（默认） | LLM 优先，失败回退规则 | 兼顾效果和可用性 | 略复杂 |

### 5.3 规则词典设计

词典以**标准术语 → 口语表达列表**的映射结构组织，优先长匹配：

```python
# query_optimizer.py:42-162
COLLOQUIAL_MAP = {
    "腹痛": ["肚子疼", "肚子痛", "肚疼", "胃痛", "心口疼", ...],
    "腹泻": ["拉肚子", "拉稀", "跑肚", "闹肚子", ...],
    "心悸": ["心慌", "心跳快", "心里发慌", "心突突", ...],
    # ... 覆盖消化、呼吸、心血管、骨骼、皮肤等 12 个系统
}
```

匹配算法：按口语表达**长度降序**遍历，优先匹配长短语（避免"肚子疼"被拆成"肚子"+"疼"），去除重叠匹配。

### 5.4 LLM 标准化 Prompt

```python
# query_optimizer.py:334-355
system_prompt = """你是一个中文医疗文本标准化助手。
规则:
1. 口语→标准: "肚子疼"→"腹痛"、"拉肚子"→"腹泻"、"发烧"→"发热"
2. 方言→标准: "打摆子"→"寒战发热"
3. 保留程度/频率/位置等修饰词
4. 不要捏造用户没说过的症状
5. 保留紧急信号（剧烈胸痛、大出血等）

输出格式: JSON
{
  "standardized_text": "...",
  "symptoms": [...],
  "severity": "轻/中/重/未知",
  "has_emergency_signals": false
}"""
```

### 5.5 优化效果

实测数据（288 个测试用例）：

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 平均置信度 | 66.01% | 69.79% | +3.78% |
| 改善比例 | — | 73.75% 的案例 | — |

## 七、DeepSeek API 配置与模型选择

### 7.1 配置方式

通过 `.env` 文件管理 API Key：

```env
DEEPSEEK_API_KEY=sk-xxxx
```

```python
# deepseek_client.py:32
load_dotenv(os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", ".env"))
```

## 八、容错与降级设计

系统在多个层面设计了容错机制：

```
                    ┌─────────────────┐
                    │  用户输入症状    │
                    └────────┬────────┘
                             │
              ┌──────────────▼──────────────┐
              │    QueryOptimizer (Layer 0)  │
              │                             │
              │  LLM 可用? ──Yes──> LLM 标准化
              │     │                      │
              │    No                      失败?
              │     │                      │
              │     └──────> 规则词典 ──────┘
              │                 │
              │              无匹配?
              │                 │
              │                 └──> 保留原始输入
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │    VectorStore (Layer 1)     │
              │                             │
              │  相似度 < 0.7? ──Yes──> 降低置信度
              │  模型未加载? ──Yes──> 延迟加载
              │  Collection 不存在? ──> ChromaDB 报错
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │   DeepSeekClient (Layer 2)  │
              │                             │
              │  API 调用成功?               │
              │    ├── Yes, JSON 合法? ──> 正常返回
              │    ├── Yes, JSON 非法? ──> 文本提取 + 兜底
              │    └── No (网络/认证异常) ──> 错误结果
              │         {"department": "服务异常", ...}
              └─────────────────────────────────────┘
```

**关键兜底策略**：

1. **查询优化层**：LLM 不可用时自动降级到规则词典；规则词典无匹配时保留原始输入
2. **检索层**：BGE-M3 模型本地部署，不依赖网络；ChromaDB 持久化，重启不丢失
3. **生成层**：JSON 解析失败时尝试从文本提取关键字段；API 完全不可用时返回明确错误信息而非崩溃

## 十一、与纯 LLM 方案的对比

| 维度 | 纯 DeepSeek LLM | RAG + DeepSeek（本项目） |
|------|----------------|------------------------|
| 知识来源 | 训练数据（可能过时/幻觉） | 8,808 条结构化知识库（可追溯） |
| 科室推荐 | 可能编造不存在的科室 | 严格限定在知识库 54 个科室内 |
| 置信度 | 无法量化 | 向量相似度 + LLM 置信度双重评估 |
| 可更新性 | 需重新训练/微调 | 增量添加数据即可 |
| 可解释性 | 黑盒推理 | 显式推理链："症状→疾病→科室" |
| 成本 | 单次推理 | 检索（免费）+ 生成（~700 tokens） |
